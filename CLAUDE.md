# 투자정보 — 프로젝트 가이드

## 시점 컨텍스트 (필독 — AI/모델 사전학습 시점 의존 금지)

**대한민국 현직 대통령: 이재명** (더불어민주당, 2025년 6월 취임)
- 윤석열 전 대통령은 2025년 4월 4일 헌법재판소 탄핵 인용으로 파면됨
- 뉴스 원문에서 "李대통령" "이 대통령" "대통령"으로만 표기된 경우 **반드시 "이재명 대통령"으로 해석**
- 시황 브리핑·종목 요약·뉴스 헤드라인 어떤 산출물에도 절대로 "윤석열 대통령" 표기 금지
- 한국 정부 = "이재명 정부" (윤석열 정부는 2025년 4월 이전 과거 사건 인용 시에만 사용 가능)

## 프로젝트 개요
한국 투자정보 대시보드. Python 백엔드(수집·스크래퍼·FastAPI 서버·푸시 봇) + React 대시보드.

## 스택
- **프론트엔드**: React 19 + Vite 8 + Tailwind v4 + Framer Motion 12 + Recharts / Lightweight Charts + React Router 7 + Firebase + Axios
- **백엔드**: Python 3.11, FastAPI, Postgres (`server/db_compat.py`의 SQLite-방언 호환 셰임 경유 — 코드 다수가 `?` 플레이스홀더·`INSERT OR IGNORE` 등 SQLite 문법으로 작성돼 있어도 실제 저장은 PG), Uvicorn
- **외부 API**: 한국투자증권(KIS), OpenDART, KRX, 네이버 증시/테마, pykrx

## 디렉터리 맵
- `dashboard/` — 프론트엔드 (React). 위젯 70+개가 `src/components/widgets`에 있음
- `dashboard/src/views` — 라우트 단위 뷰
- `server/` — FastAPI 서버 (`server/db/connections.py`가 PG 단일 진입점)
- `collectors/` — 외부 API 원천 수집기
- `scrapers/` — HTML 스크래퍼
- `references/` — KIS 기준 참조 구현. `모듈.py` ↔ `chk_모듈.py` 페어 관습
- `scripts/` — 일회성/정기 스크립트 (cron·APScheduler에서 호출)
- `calculators/` — 기술 지표
- `persona/` — AI 인격/요약
- `scratch/` — 실험 코드 (정리 대상)

## 프론트엔드 디자인 원칙 — 하네스 평가 기준

생성된 UI는 **design-evaluator** 서브에이전트가 4축으로 평가. 가중치:

| 축 | 가중치 | 기준 |
|---|---|---|
| 디자인 퀄리티 | 35% | 색/간격/타이포 위계의 전체 일관성 |
| 오리지널리티 | 30% | AI 전형 패턴 회피 (아래 블랙리스트) |
| 크래프트 | 20% | 반응형, 다크모드, 접근성, 애니메이션 적정성 |
| 펑셔널리티 | 15% | 로딩/에러/빈 상태, 키보드, 터치 타겟 |

### AI 전형 패턴 블랙리스트 (즉시 감점)
- 보라↔파랑 그라디언트(`from-purple-* to-indigo-*`, `from-violet-* to-blue-*`) 배경
- 흰 카드 + 둥근 모서리(`rounded-2xl`) + 그림자 조합 남발
- 이모지를 강조 수단으로 반복 (🚀 ✨ 🎉 등)
- 모든 제목에 그라디언트 텍스트
- 센터 정렬된 히어로 + 큰 버튼 2개 템플릿
- 동일 톤의 블루 계열만으로 모든 상태(정보/성공/경고) 표현

### 위젯 일관 규칙
- 카드 패딩: `p-5` (모바일) / `p-6` (md+) 통일. `.toss-card` 유틸이 이미 반영.
- 헤더: **`DetailWidgetHeader`** — `english` + `accent` + `icon`(lucide) + `title` + `subtitle`. 이모지 string은 자동 무시됨.
- 다크모드 페어: 모든 텍스트에 `dark:text-*` 명시
- 데이터 없음 상태: 빈 div 금지, 명시적 placeholder 또는 skeleton

### v2.4 디자인 시스템 (상세 페이지 레이어 참고)

**Shell 토큰** (`index.css @theme`):
- `--color-shell-bg: #FFFFFF` (body, 라이트)
- `--color-shell-surface: #FFFFFF` (카드 — body와 동일, border로만 구분)
- `--color-shell-inline: #F9FAFB` (강조 블록 — Trio row 배경)
- `--color-shell-border: #E5E8EB`
- 다크 페어: `-dark` suffix (`#181C23 / #252930 / #2E3138 / #31363F`)

**`.toss-card` 유틸**:
```
bg-[shell-surface] + border 1px warm + rounded-[14px] + shadow 0
```
shadow는 라이트·다크 모두 **0**. 카드 간 구분은 `border` 1px만.

**Trio row 패턴** (Hero + 대다수 위젯의 핵심 블록):
```jsx
<div className="rounded-[14px] overflow-hidden bg-[var(--color-shell-inline)]
  grid grid-cols-1 sm:grid-cols-3 divide-y sm:divide-y-0 sm:divide-x
  divide-[var(--color-shell-border)]">
  <TrioCell stripeClass="bg-toss-red" label="..." value="..." sub="..." />
  ...
</div>
```
- 각 row 좌측 3px 컬러 stripe
- `font-mono tabular-nums` 수치
- 라벨 11px bold uppercase, 수치 20-26px bold, 서브 11px bold

**시그널 컬러 토큰** (의미 있는 강조만):
- `toss-red #F04452` — 상승·위험
- `toss-blue #3182F6` — 하락·정보
- `toss-orange #FF6B35` — 경고·개인
- `toss-purple #7238FF` — 기관·기타
- 나머지는 `toss-gray-*` 스케일

**타이포 스케일** (Hero가 최대, 위젯은 Hero 내부 Trio보다 작게):
- Hero 현재가: `clamp(56px, 8vw, 112px)` — 뷰 전체 최대
- Hero 판정: 24-32px semibold
- 위젯 제목(DetailWidgetHeader title): 18-20px bold
- Trio 수치: 20-26px bold
- 본문·라벨: 11-14px bold/medium
- `font-black`은 **사용 금지** (font-bold로 통일, 강조는 크기·컬러로)

**font-weight 위계 (2026-04-22 사용자피드백 "굵은 글씨 눈 아파"):**
| 레벨 | 용도 | weight |
|---|---|---|
| 1 | Hero 현재가·섹션 제목·판정 문장 | `font-bold` |
| 2 | 위젯 제목(DetailWidgetHeader title)·Trio 수치·주요 숫자(font-mono tabular-nums) | `font-bold` |
| 3 | 위젯 내 헤드라인·배지·uppercase 라벨 | `font-semibold` (예외적) |
| 4 | 서브 라벨·본문 paragraph·설명 텍스트 | **`font-medium`** (기본) |
| 5 | 발행 시각·출처·subdued 메타 | `font-medium` 유지 (색상으로 구분) |

**금지**:
- 11-13px 본문·라벨에 `font-bold` 사용 (레벨 4인데 weight 올라가면 시각 피로)
- 한 카드 안에 `font-bold` 텍스트 5개 초과 (위계 붕괴 → 전부 외치기)
- 주요 수치와 같은 크기의 라벨을 `font-bold`로 (수치 강조 효과 소멸)

**허용**:
- 수치는 `font-mono tabular-nums font-bold`로 강조 유지 (숫자는 굵어도 눈 피로 덜함)
- 배지 내 글씨는 작은 사이즈(9-10px) + `font-bold` OK (pill 배경과 대비)

**카피 톤** (v2.1 토스-light):
- Hero 판정 문장: 데이터 앵커 + 짧은 친근 ("외국인이 7일 연속 매집 중이에요. 누적 +2,340억.")
- 빈 상태·로딩: 친근 허용 ("데이터 수집 중이에요")
- 이모지는 **전역 금지** (DetailWidgetHeader가 string 이모지 자동 무시)
- 반복 라임 ("~했어요/~몰렸어요/~버티고 있어요") 과사용 금지

**섹션 구조** (StockDetailView):

## 워크플로우

### 3-에이전트 루프 (Planner → Generator → Evaluator)

**언제 어느 루프를 쓰나:**
- **개선 루프 (Generator↔Evaluator)**: 이미 방향이 정해진 위젯의 결함 수정. 2~5회로 수렴.
- **재설계 루프 (Planner→Generator↔Evaluator)**: 뷰 전체가 별로일 때, 구조·위계·정체성 재결정 필요할 때.
- **카운슬 루프 (4 비평가 + 종합자)**: 단일 평가자로 잡히지 않는 고수준 이슈 — 주린이 이해도·창의성·의사결정 가치가 동시에 의심될 때. 큰 마일스톤·방향 점검에만.

#### 개선 루프 (2인조)
1. 메인 에이전트(Generator)가 위젯/뷰 수정
2. `/evaluate <widget-path-or-route>` → `design-evaluator` 실행
3. Playwright MCP 실측 + 4축 점수 + 레퍼런스 대비 서술 + 결함 리포트
4. 실패 축 있으면 피드백대로 메인이 수정, 다시 평가
5. 단일 위젯 3~5회, 전체 페이지 5~15회가 경험적 수렴 구간

#### 재설계 루프 (3인조)
1. `/plan <target-view> -- <purpose>` → `design-planner` 실행
2. Planner가 `scratch/plans/<timestamp>_<topic>.md`에 스펙 저장
   - 정체성, Hero 전략, 섹션 구조, 제거 리스트, 카피 톤, Generator 작업 큐, Evaluator 체크리스트
3. 사용자가 스펙 승인 → Generator가 작업 큐 순서대로 구현
4. 각 작업 단위 후 `/evaluate`로 검증, 통과 시 다음 단위
5. 모든 큐 완료 후 Evaluator "레퍼런스 대비 서술"을 읽고 다음 재설계 루프 필요 여부 판단

#### 카운슬 루프 (4 비평가 + 종합자)
1. `/council <widget-path-or-route>` 호출
2. **Round 1 (병렬)**: 4명이 독립적으로 평가
   - `design-evaluator` — 4축 크래프트·블랙리스트·반응형 (기계 체크)
   - `design-critic-newbie` — 주린이 페르소나. "이 용어 뭔 뜻?" "뭘 해야 되죠?" 이해도·행동 유도
   - `design-critic-creative` — 기시감·오리지널리티. 대안 아이디어 최소 2개 제안
   - `design-critic-analyst` — 10년차 투자자. 의사결정 가치·수치 맥락·한국 시장 컨텍스트
3. **Round 2 (종합)**: `design-council-synthesizer`가 4개 리포트를 **합의 / 충돌 / 단독 주장**으로 정리 + 우선순위 액션 리스트
4. 메인이 합의 이슈부터 수정 착수, 충돌 이슈는 사용자 결정 요청
5. 재수정 후 일반적으로 `/evaluate`로 빠르게 확인 (카운슬 재호출은 큰 구조 변화 있을 때만)

**카운슬 vs 단일 평가**: 카운슬은 4배 토큰·시간. 일반 개선은 `/evaluate` 유지. 카운슬은 (a) 뷰 전체 점검 (b) "전체적으로 별로" 판정 시 어느 축이 문제인지 진단 (c) 주린이/전문가 충돌 해결이 필요할 때.

### 평가자 편향 회피 — 감점 기반 vs 가점 기반
- 4축 점수(감점 기반): 블랙리스트 위반·다크 페어 누락·터치 타겟 등 **기계적 체크** — 안전한 평균에 수렴
- 레퍼런스 대비 서술(가점 기반): "토스/로빈후드/블룸버그 대비 어디가 나은가·밀리는가" — 고수준 매력 체크
- 감점만 피해서 PASS 받아도 "전체적으로 별로"면 재설계 루프로 전환하는 신호

### 빌드 확인
- `cd dashboard && npm run build` — 타입/번들 에러 검출. UI 검증은 빌드 성공 후 evaluator로.

## AI 요약 루틴 (Claude Code가 직접 수행)

Gemini/외부 AI API를 호출하지 않는다. **Claude Code 에이전트 자체가** 주기적으로 깨어나 데이터를 읽고, 스스로 요약을 만들어 DB에 쓴다. 시황 브리핑(`market_briefings`)과 동일 패턴.

### 대상 루틴 2종

| 루틴 | 저장 테이블 | 프론트 노출 |
|---|---|---|
| 종목 일일 한 줄 요약 | `stock_daily_summary` | StockSummaryHero Hero 서브 카드 |
| 테마 맥락 해설 | `theme_daily_context` | HotThemeRaceWidget 카드 확장 |

### 루틴 실행 흐름 (Claude Code가 따를 스텝)

1. **타겟 확정**
   - `python scripts/ai_summary_targets.py --kind stock --limit 50 --skip-done` → JSON 배열 (아직 오늘 요약이 없는 종목)
   - `python scripts/ai_summary_targets.py --kind theme --limit 20 --skip-done`

2. **컨텍스트 읽기** (타겟 하나마다)
   - `python scripts/ai_summary_context.py --kind stock --code 000660` → 가격/수급 5일/기술지표/DART/뉴스 JSON
   - `python scripts/ai_summary_context.py --kind theme --theme "2차전지"` → 평균 등락률/종목 Top3/관련 뉴스 JSON

3. **스스로 요약 생성** (Claude Code의 thinking)

4. **DB 기록**
   - 종목: `python scripts/ai_summary_upsert.py --kind stock --code ... --one-liner "..." --drivers "..." --tone up --used-signals "price,flow,tech,news"`
   - 테마: `python scripts/ai_summary_upsert.py --kind theme --theme "..." --context "..." --drivers "..." --tone up`
   - 실패: `--status error --error "no_data"` (재시도 대상으로 남음)
   - `updated_at`은 upsert 스크립트가 실행 시각으로 자동 기록 → 프론트에서 "발행 시각"으로 노출

### 카피 규약 (주린이 타겟)

**공통**
- 말 끝 `-요/-에요` 자연스럽게. 명령·단정 금지.
- 감정 부사 금지: `살짝`, `벌써`, `아직`, `무려`, `엄청`
- 투자 권유·추천 단어 금지: `매수`, `매도`, `사세요`, `팔아야`. 허용: `관찰`, `주의`, `참고`.
- 숫자는 컨텍스트에 있는 값만 사용. 근거 없으면 적지 말 것.
- 전문 용어(이평선·이격도·신고가·공매도 등)는 써도 OK — 프론트의 `TermTooltip`이 풀이 제공.

**종목 `one_liner` (25-60자)**
- 형식: `<동력 1개> + <동력 2개> → <움직임>`
- 예시
  - `외국인 3일 연속 매수에 +3.2% 올라 60일 신고가를 찍었어요.`
  - `MA5 이격도 -8% 진입 + 개인 반발 매수로 보합권 회복 중이에요.`
  - `유상증자 공시 후 -7% 급락했어요. 기존 주주 지분 희석이 반영되고 있어요.`

**종목 `one_liner` — 연재 연속성 (2026-06-22 신설, brief "약속 이행" 규약의 종목판)**

StockStorylineTimeline 위젯이 일별 one_liner 를 그대로 타임라인으로 보여줘서, one_liner 자체가 "연재"가 된다. `ai_summary_context.py` 가 주는 `prev_summary`(어제 one_liner) 와 `dart_recent`(임팩트·강도) 를 **반드시 활용**:

1. **어제 이야기를 이어라** — `prev_summary.one_liner` 를 읽고, 흐름이 이어지면 연결 표현.
   - ✅ `3일 전부터 들어온 외국인 매집이 오늘도 이어져 +2%, 누적 5일째예요.`
   - ✅ `어제 말한 공매도 부담이 오늘 해소되며 +4% 반등했어요.`
   - ❌ 매일 단절된 스냅샷 (`+2% 올랐어요`만) — 어제와 무관하게 새로 쓰기 금지
   - 흐름이 **바뀌었으면** 그 전환을 짚어라: `어제까지 외국인 매수였는데 오늘 매도 전환되며 -3%.`

2. **그날 강한 공시·뉴스가 있으면 반드시 엮어라** — `dart_recent` 중 `impact_strength >= 2` 또는 `news_today` 의 큰 재료.
   - ✅ `어제 300억 수주 공시(매출 12%)가 오늘 +5%로 반영됐어요.`
   - ✅ `유상증자 공시로 -7% 급락, 지분 8% 희석이 부담이에요.`
   - 강한 공시가 있는데 one_liner 가 수급 얘기만 하면 누락 위반.

3. **여러 이유를 엮어라** — "외국인 며칠째" 단일 동력만 반복 금지. 수급 + 공시/뉴스 + 기술지표 중 그날 진짜 동력 2개를 조합 (형식의 `<동력 1개> + <동력 2개>` 충실히).

**종목 `drivers` (1-3개 키워드 배열)**
- 명사구 중심. `외국인 매집`, `60일 신고가`, `유상증자 공시`, `HBM 호재`
- 형용사·수식어 최소. `조금 강한 매수세` ❌ → `외국인 매집` ✅

**종목 `tone` (enum)**
- `up`: 상승 동력 우세
- `down`: 하락 동력 우세
- `neutral`: 방향성 불명 / 보합권
- `risk`: 공시·공매도 급증 등 주의 플래그

**종목 `used_signals` (enum 배열)**
- `price`, `flow`, `tech`, `dart`, `news`, `short` 중 **실제 근거로 쓴 것만**. 팩트에 있어도 쓰지 않았으면 제외.

**테마 `context` (40-90자, 1-2문장)**
- 형식: `<외부 뉴스 동력> + <내부 종목 주도> → <평균 등락률>`
- 예시
  - `해외 배터리 CAPEX 발표와 LFP 전환 뉴스가 겹치며 +4.2% 상승. 엘앤에프가 +8%로 앞장섰어요.`
  - `뚜렷한 뉴스 동력 없이 단독 수급 유입으로 +2%. 특정 재료 확인은 어렵습니다.`
  - `미국 상호관세 발표 우려로 -3.1% 하락. 삼성SDI·LG에너지솔루션이 낙폭 주도.`

**테마 `drivers` (1-3개)**

**테마 `tone`**: `up` / `down` / `neutral`

## 시황 브리핑 — "체크 포인트" bullet 카피 규약 (2026-05-11 강화)

```json
"action_chip": {

```

### body 카피 규약 (30-60자 한 문장)

**형식**: `<관찰 대상 + 임계값> — <임계값 의미>`

- `"기관 매도가 더 커진 줄다리기 장이에요"` (진단 — 이건 bullet 본문에 쓰는 말)

### 중복 방지 (필수)

- 체크 포인트 **bullet 본문** = 상황 진단 + 맥락 (2-3문장)
- `action_chip.body` = 임계값·신호 한 줄 (1문장)
- **둘이 같은 말 반복 금지**. 본문은 "지금 무슨 일이 일어나고 있나", chip 은 "그래서 뭘 봐야 하나".

- 시장이 평이하고 특별한 단서 없는 날 — 거짓 임계값 만들지 말 것
- 임계값이 불명확한 상황 ("두고 봐야 함", "지켜봐야 함" 같은 모호함)
- 이미 결과가 나온 사건 (체크포인트는 미래 관찰용)

## 주린이 가이드 3필드 — newbie_summary / macro_context / newbie_guide (2026-05-29 신설)

모든 brief slot 의 `summary_structured` 에 **선택 3필드**를 emit. 프론트(홈 MarketMoodHero + /brief/today)가 "주린이 가이드" 박스로 노출. 전문 bullets 와 별개로, **진짜 입문자가 읽고 "그래서 뭐?"가 안 남게** 하는 번역 레이어.

```json
"summary_structured": {
  "headline": "...", "bullets": [4 고정], "closing": "...", "action_chip": {...},
  "newbie_summary": "오늘 장을 초보 말로 한 줄 + 큰일인지 아닌지",
  "macro_context": "며칠~몇 주 단위 큰 흐름 (외국인 N일 연속 매도 등) — 있을 때만",
  "newbie_guide": "오늘 같은 날 학습·관찰 프레이밍 — 투자 권유 절대 금지"
}
```

### newbie_summary ("오늘 한마디", 50-110자)

주린이가 가장 궁금한 **"이거 큰일이야 아니야?"** 에 답. 헤드라인의 친근한 번역 + 평가.

- 형식: `<오늘 무슨 일 — 쉬운 말> + <큰일인지 아닌지 판정 + 근거>`
- **비교 베이스라인 의무** (CLAUDE.md "비교 베이스라인 의무화" 연장): "-3.84%는 가끔 나오는 큰 변동", "이 정도는 평범한 조정"
- 좋은 예: "어제 +8% 폭등한 게 오늘 일부 되돌려진 거예요. 코스피 -3.84%는 가끔 나오는 큰 변동이지만, 새 악재가 터진 게 아니라 어제 많이 오른 반도체가 쉬어가는 흐름이라 패닉할 날은 아니에요."
- 나쁜 예: "코스피가 하락했어요." (베이스라인·판정 0)

### macro_context ("요즘 흐름", 40-100자) — **있을 때만**

**오늘 하루만 보면 안 보이는 며칠~몇 주 단위 큰 관점.** `brief_context_inject.multiday_flow` 데이터 활용.

- 근거 데이터:
  - `multiday_flow.foreign.streak_days` + `streak_dir` — "외국인 N일 연속 매도/매수"
  - `multiday_flow.foreign.sum_5d_uk` / `sum_20d_uk` — 5일·20일 누적
  - **streak 가 길면 연속일수**로, streak 짧지만 **누적 크면 누적**으로 (둘 중 진짜 스토리 선택)
    - 예: streak 1일이어도 sum_5d -17조면 → "외국인이 5일 동안 17조 던진 큰 매도세"
  - 환율 레짐(`fx_macros`), 코스피 추세(`recent_briefs` 며칠 흐름)도 가능
- **emit 안 하는 날**: 며칠 흐름이 평이하거나 특별한 추세 없을 때 (거짓 강조 금지). 매일 고정으로 "외국인 매도 중" 박지 말 것 — 그날 진짜 의미 있을 때만.
- 좋은 예: "외국인은 5일 동안 17조를 순매도하며 매도세가 무거워요. 다만 기관이 받아내면서 지수는 버티는 중이라, 외국인이 돌아오는지가 반등의 열쇠예요."
- 나쁜 예: "외국인이 팔고 있어요." (기간·규모·맥락 0)

### newbie_guide ("오늘은 이런 날", 50-120자) — **투자 권유 절대 금지**

**"그래서 나는 뭘 해야 해?"** 에 답하되, 매매가 아닌 **학습·관찰·마음가짐** 프레이밍.

- 🚫 **금지**: 매수·매도·사세요·팔아야·추천·관망 추천·진입·손절. (CLAUDE.md AI 요약 금지 규약 연장)
- ✅ **허용**: 공부하기 좋은 날 / 한 박자 늦게 / 침착하게 / 관심 종목 정리 / 업종별로 살펴보세요 / 눈여겨보세요
- 시장 상황별 톤:
  - 변동 큰 날 → "출렁임 큰 날이에요. 급하게 따라가기보다 한 박자 늦게, 왜 움직였는지 공부하기 좋은 날이에요."
  - 평온한 날 → "큰 사건 없이 잠잠한 날이에요. 관심 종목 펀더멘털 정리해 보기 좋은 날이에요."
  - 급락 날 → "많이 빠진 날이에요. 패닉보다 왜 빠졌는지 차분히 보고, 평소 관심 종목이 같이 빠졌는지 살펴보세요."
  - 양극화 날 → "오르는 종목·빠지는 종목이 동시에 많아요. 시장 전체보다 업종별로 갈리는 흐름을 살펴보세요."
- 나쁜 예: "지금 매수 타이밍이에요" / "관망을 추천해요" (권유) — 즉시 위반

### 3필드 공통

- 이모지 금지, `-요/-에요`, 괄호 풀이 금지 (TermTooltip 자동)
- newbie_summary 는 거의 매번 emit (오늘 장 요약은 항상 가능). macro_context·newbie_guide 는 의미 있을 때만.
- 셋 다 없으면 프론트가 박스 자체를 안 그림 — 빈 박스 안 생김.

## 홈 헤드라인 — 중요도 우선 동적 선택 (2026-06-11 신설)

**사용자 피드백 인용**: "오라클이 빅비트면 그건 꽤 중요한 내용인데 홈에선 코스피지수랑 상한가 이야기만 함. 홈에 나오는 내용은 중요도를 생각해서 유동적으로."

홈 `MarketMoodHero` 는 `summary_structured.headline` 을 화면 최상단에 그대로 노출. 즉 **헤드라인 = 그날 가장 중요한 한 가지**. 매번 "코스피 ±X% + 상한가 N종목" 고정 템플릿으로 채우지 말 것 — 더 큰 사건이 있으면 그게 먼저.

### 헤드라인 중요도 우선순위 (위에서부터, 그날 발생한 가장 높은 항목이 헤드라인)
1. **시장 정지·급변**: 사이드카·서킷브레이커·지수 신고가/신저가 경신·미국↔한국 디커플링
2. **빅 이벤트 결과**: 미국 빅테크 실적 빅비트/쇼크(오라클·엔비디아·브로드컴·마이크론·AMAT·TSMC·애플 등 — 한국 반도체·AI 직접 영향), FOMC·CPI·PPI 서프라이즈, 대형 IPO(스페이스X)
3. **극단 수급**: 외인·기관 ±2조 이상 폭매수/폭매도
4. **지수 큰 변동**: ±1.5% 이상
5. **평이한 날**: 지수 흐름 + 주도 테마 + breadth(상한가/급등 개수)

→ 2번이 그날 최대 사건이면 **지수·상한가보다 먼저** 헤드라인에. 예: "오라클 클라우드 빅비트로 AI 인프라株 강세 — 오늘 SK하이닉스·삼성전자 우호적이에요."

### 미국 빅테크 실적 = 헤드라인 후보 (능동 확인 의무)
빅테크 실적은 매크로 캘린더에 없음. pre/morning brief 는 ① `brief_context_inject.upcoming_calendar.earnings` ② 직전 evening/post 예고 ③ WebSearch 로 **간밤 빅테크 실적 결과를 능동 확인**하고, 빅비트/쇼크면 헤드라인 또는 첫 bullet 으로. 결과 표기는 "어닝 결과 한국어 표기 규칙"(빅비트=상회 / 쇼크=하회) 준수.

### 금지
- 그날 더 큰 사건(빅테크 실적·매크로 서프라이즈·사이드카)이 있는데 "코스피 ±X%·상한가 N종목"으로 헤드라인 때우기
- breadth(상한가·급등 개수)를 평이하지 않은 날 헤드라인 메인으로 (평이한 날에만 보조로)

## brief 카피 톤 — "왜 그랬나" 인과 분석 의무 (2026-06-02 강화)

**사용자 피드백 인용**: "그냥 장 상황 그대로 읽어주는건 좀 아닌듯. 떨어졌으면 왜 떨어졌을까 하는 추측이나 그런게 필요하지. 이건 좀 너무 아무 내용이 없다."

### 금지 패턴 — 단순 사실 나열

❌ "외국인, 코스피 매도세 지속"
❌ "기관 매수 우위"
❌ "지수 약세 흐름"
❌ "수급 회복 기대"

위 패턴은 정보가 0. headline · bullet · closing 어느 위치에도 단독 등장 금지.

### 필수 — 인과 분석 1줄 동반

모든 본문 (headline + bullets + closing) 의 사실 진술 옆에는 **"왜"** 가 붙어야 한다. 추측이라도 OK — 단 **데이터 anchor** 명시.

✅ 형식 — `<사실> + <왜 — 데이터 anchor>`:
- "외인 매도 -8,000억 — 반도체 기판(-9.4%) · MLCC(-7.7%) 약세가 SK하이닉스 · 삼성전자에 직접 영향"
- "미국이 +0.8% 올랐는데 코스피만 -1.5% 빠진 디커플링 — 환율 1,365원 상승 + 6/12 스페이스X IPO 자금 회수 + 브로드컴 실적 직전 차익실현 3중"
- "보험 · 통신 방어주가 +3% 강세 — AI 자금이 거품 부담 인식하고 디펜시브 갈아탔어요"

### 인과 anchor — `brief_context_inject` 의 다음 데이터 활용 의무

`scripts/brief_context_inject.py` 가 emit 하는 분석 데이터를 brief 본문 인과 추측에 의무적으로 활용:

| 데이터 | 인과 분석 후보 |
|---|---|
| `today_snapshot.themes.falling_top5` | 약세 테마 → 지수 끌어내린 원인 |
| `today_snapshot.top_movers.losers_top5` | 시총 큰 하락 종목 → 외인 매도 주체 식별 |
| `overnight_us.us_indices` | 미국 ↑↓ vs 한국 ↑↓ → 디커플링 / 동조화 분석 |
| `overnight_us.us_bigtech` | NVDA · AVGO 어제 등락 → 한국 반도체 영향 예측 |
| `overnight_us.overnight_events` | 어제 미국 PPI · CPI · FOMC → 한국 외인 수급 변화 |
| `fx_macros.fx.usd_krw` | 환율 변동 → 외인 매수/매도 방향 |
| `today_disclosures_down` | 오늘 악재 공시 → 특정 종목 급락 원인 |
| `mega_events` (예: 스페이스X IPO D-N) | 글로벌 자본 흐름 영향 → 한국 외인 사전 매도 |
| `today_snapshot.flow_live` | KIS 5분 폴링 가장 신선 누적 수급 — brief.flow_today 가 stale 일 때 우선 |

### 분석 깊이 — 1줄 추측은 OK, 0줄은 금지

추측 카피는 "~로 보여요" / "~ 영향 가능" / "~ 직전 차익실현으로 추정" 등 **추측 어미** 명시 OK. 데이터 anchor 만 있으면 추측이라도 가치 있음.

❌ "그냥 약세였어요" (anchor · 추측 둘 다 0)
✅ "약세 — 반도체 기판 테마(-9.4%) 가 외인 매도 -8천억 주도한 것으로 보여요" (anchor 2개, 추측 어미)

### 다중 원인 — 2-3 anchor 까지 OK

큰 변동 (지수 ±1% 이상 / 외인 ±2조 이상) 은 단일 원인보단 **다중 원인** 카피가 정확:

✅ "코스피 -1.5% 차익실현 — (1) 환율 1,365 으로 상승 (2) AVGO 어제 -3% 로 반도체 약세 (3) 6/12 SPCX IPO 직전 자금 회수, 3중 영향"

### 슬롯별 인과 anchor 우선순위

| 슬롯 | 우선 활용 anchor |
|---|---|
| pre (06:30) | overnight_us + fx_macros + 매크로 결과 |
| morning (11:30) | flow_live + themes.falling_top5 + losers_top5 |
| afternoon (14:00) | morning 대비 변화 + flow_live 변화 + themes 교체 |
| post (15:40) | 오늘 마감 총괄 + 외인 누적 final + 디커플링 분석 |
| evening (20:00) | 오늘 마감 + 미국 프리마켓 + 이번 주 핵심 이벤트 prep |

### 수급 데이터 — 가장 신선 source 우선

기존: brief 작성 시점 `fetch_market_flow.py` 결과를 `flow_today` 에 저장 → brief emit 후 freeze.
**개선**: brief 작성 직전에 `brief_context_inject.today_snapshot.flow_live` (KIS 5분 폴링 최신값) 도 같이 확인.
- 두 값이 30분 이상 다르면 flow_live 가 더 신선 → 그 값으로 `flow_today` 채움
- 본문 인과 분석에도 flow_live 시점 명시 ("11:55 기준 외인 -5조 4천억 — 5분 전 KIS 데이터")

### 자동 검증 (추후) — anchor 0개 본문 거부

후속 작업: `ai_briefing_upsert.py` 에 본문 검증 추가:
- `summary` 안에 데이터 anchor 패턴 (퍼센트 · 금액 · 종목명 · 테마명) 최소 3개 검출 못하면 **reject**
- 단순 정성 문구만 있으면 "왜 분석 부족" 으로 분류 → 재작성 요청

## brief 연속성 — "약속한 이벤트는 다음 brief에서 반드시 이행" (2026-05-15 강화)

evening/post brief 가 다음 brief 시점에 발표될 이벤트를 명시적으로 예고했다면(예: "오늘 밤 21:30 미국 소매판매 발표", "새벽 5:30 어플라이드 머티리얼즈 실적", "내일 06:30 정리해 드릴게요"), 다음 pre/morning brief 는 그 이벤트들의 결과를 **반드시 본문 첫 문장 또는 두 번째 문장에 cover** 해야 한다.

### 절차

1. **brief 작성 시작 전** — 직전 evening/post brief 본문을 DB에서 읽어 "예고된 이벤트" 추출
   ```sql
   SELECT summary FROM market_briefings
   WHERE briefing_date = '<어제>' AND slot IN ('evening', 'post')
   ORDER BY slot_time DESC LIMIT 1
   ```
2. 예고 이벤트 추출 키워드: `발표`, `실적`, `정리해 드릴게요`, `좌우할 수 있어요`, `결과는`, `내일 06:30/장전 브리핑에서`
3. 매크로 이벤트는 `collectors/macro_collector.py::MacroCollector.get_us_overnight_events()` 로 결과 확보
4. 미국 빅테크 실적(AMAT·NVDA·TSMC·MU·AVGO 등)은 매크로 캘린더에 안 들어옴 — `WebSearch` 또는 `news_events` 테이블 검색으로 결과 확보
5. **결과를 못 가져온 경우 절대 침묵 금지** — "AMAT 결과는 아직 확인되지 않았어요, morning brief 에서 다시 짚어 드릴게요" 같이 솔직히 짚되 반드시 언급

### 금지

- 예고한 이벤트의 결과를 cover 안 하고 다른 주제로 brief 전체 채우기 (사용자 신뢰 깨짐)
- "어젯밤 미국은 사상 최고치" 같은 일반 시장 요약으로 약속한 구체 이벤트 대체
- 예고된 종목(SK하이닉스·삼성전자) 관찰점을 누락한 채 "외인 선물 흐름" 같은 일반 단서로 대체

### 좋은 예 (2026-05-15 핫픽스 적용 본문)

"어제 약속드린 두 이벤트 정리해 드릴게요. 미국 4월 소매판매는 헤드라인 +0.5%로 예상 부합이었지만 자동차 제외 +0.7%·컨트롤 그룹 +0.5%는 예상 상회로 소비 견조 확인이에요. (...) 어플라이드 머티리얼즈는 매출 79억 달러 사상 최고치, EPS 3.51달러로 컨센서스 2.71달러 크게 상회, 2026 장비 매출 가이던스 +30% 이상에 HBM·DRAM·첨단패키징 강세를 분명히 했어요 — SK하이닉스·삼성전자 출발에 우호적 환경이에요."

→ evening 에서 약속한 (a) 소매판매, (b) AMAT 실적, (c) SK하이닉스·삼성전자 관찰점 3가지 모두 첫 단락에 cover.

## 전날 미국 매크로 이벤트 — morning/pre brief 의무 (2026-05-14)

morning(장 시작 전) / pre(장 시작 직전) brief 작성 시 **전날 KST 밤~오늘 새벽 미국에서 발표된 매크로 이벤트**를 빠짐없이 다뤄야 한다. 사용자 피드백 인용: "아침 브리핑에서 전날 미국에서 PPI 있었을 수도 있고 뭔가 특별한 이벤트들이 있었을 수도 있는데 그거에 대해서 안 다루네."

### 데이터 소스
`collectors/macro_collector.py::get_all_macros()` 의 `us_overnight_events` 필드 (Trading Economics 캘린더 파싱). 각 이벤트 dict 구조:
```
{
  "title": "PPI MoM",
  "reference": "APR",          # 참조 기간 (4월 발표분)
  "kst": "2026-05-14 01:30",   # 한국 시간
  "actual": "1.4%",            # 실제 발표치
  "forecast": "0.4%",          # 컨센서스
  "previous": "0.7%",          # 직전치
  "previous_revised_from": null,
  "importance": "high",        # high / medium / low
  "surprise": "above"          # above / in_line / below (실제 vs 예상)
}
```

기본은 `min_importance="medium"` 이상만 + 직전 30시간. high 이벤트는 무조건 brief 에 언급.

### 카피 패턴

**문장 1 — 가장 큰 surprise 이벤트 강조 (있을 때만)**
- 형식: `<지표명> <실제>로 발표 — 예상 <예상> 대비 크게 <상회/하회>. <한국 시장 함의 1줄>.`
- 예시
  - "미국 PPI(4월) 1.4%로 발표 — 예상 0.4% 대비 크게 상회. 인플레 우려로 코스피 외인 매도 주의."
  - "CPI(3월) 2.8%로 예상 부합. Fed 6월 인하 기대 유지."
  - "Fed 의장 발언 — 추가 인하 신중. 원달러 환율 1,365원으로 상승."

**문장 2 — 오늘 한국 장에서 볼 관찰점**
- 추상 "주목" 금지 → 구체 관찰 대상
- 예시
  - "9시 30분 외국인 누적 매수가 -1000억 안쪽으로 멈추는지 확인해 주세요."
  - "반도체·금융주가 미국 동일 섹터(-2% / +1%) 와 같은 방향으로 흐르는지 봐 주세요."

### 시장 상황별 보강 패턴 (기존 8종에 추가)

```
[전날 미국 매크로 surprise]
"미국 PPI 1.4%로 예상 0.4% 대비 크게 상회 — 인플레 재점화 우려.
 9시 30분 외국인 매도 시작이 -2000억을 넘기는지 한 번 봐 주세요."

[Fed 인사 발언]
"어젯밤 Fed 의장이 N월 인하 신중 발언 — 시장 반응은 S&P -0.5% 약세.
 코스피 금융주가 미국 은행주(-1.2%) 와 같이 약세로 출발하는지 확인해 주세요."

[빅 이벤트 부재]
"간밤 미국에서 주목할 발표는 없었어요. 평이한 아침이에요.
 거래대금 상위가 어제와 비슷한 종목 위주인지 한 번만 체크해 주세요."
```

### 출력 규칙

- **반드시 다뤄야**: `importance == "high"` 이벤트가 1개 이상이면 brief 첫 bullet 에 진입.
- **선택**: `importance == "medium"` 만 있으면 "체크 포인트" bullet 안 짧게 한 줄.
- **인용 금지**: 영문 원문 그대로 인용 X. 한국어 지표명(예: "생산자물가지수 PPI")으로.
- **이모지·괄호 부가설명 금지** (기존 규칙 연장).
- **숫자 단위 명시**: "PPI 1.4%" / "비농업고용 +25만" / "원유 재고 -430만 배럴"

## 어닝 결과 표기 + brief 슬롯 간 사실 일관성 (2026-05-22 강화)

**발단**: 2026-05-21 morning brief 가 NVDA 어닝을 "어닝쇼크"로 표기. 같은 사건을 pre/post/evening 은 "빅비트·상회"로 정확히 적었는데 morning 만 정반대. 사용자 피드백 인용: "어닝 서프라이즈를 어닝 쇼크라고 하던데 왜 그런거야?"

### 어닝 결과 한국어 표기 규칙

| 실적 vs 컨센서스 | 한국어 표기 | 영문 매핑 |
|---|---|---|
| 매출·EPS 모두 **상회** + 가이던스 상향 | **빅비트** / **어닝 서프라이즈** | beat / earnings beat / surprise to the upside |
| 매출·EPS 중 일부 상회 | **부분 상회** / **혼조** | mixed |
| 매출·EPS 모두 부합 | **컨센서스 부합** | in line |
| 매출·EPS **하회** (또는 가이던스 큰 폭 하향) | **어닝 쇼크** / **어닝 미스** | earnings miss / shock / disappointment |

**원칙**:
- "쇼크" = **부정적 결과**만. 상회를 쇼크로 적는 건 사실관계 오류.
- 주가가 빠지더라도 실적 자체가 상회면 "빅비트인데 주가는 차익실현으로 -3%" 로 분리해 표기. 결과와 주가 반응을 섞지 말 것.
- 한국 시장 관례 "어닝 서프라이즈 = 긍정적" 을 따른다 (월가의 surprise 는 중립적인 양방향이지만, 한국 일반 독자에게는 긍정적 의미로 굳어짐).

### brief 슬롯 간 사실관계 일관성 (필수)

**원칙**: 같은 사건의 핵심 표기 (어닝 결과·수치·판정) 는 모든 슬롯에서 동일해야 한다.

**절차**:
1. brief 작성 시작 전 — 같은 briefing_date 의 이전 슬롯 본문을 DB 에서 읽기
   ```sql
   SELECT slot, summary FROM market_briefings
   WHERE briefing_date = ? AND slot_time < ?
   ORDER BY slot_time DESC LIMIT 3
   ```
2. 동일 사건(엔비디아 실적·FOMC·지표 발표 등)이 이전 슬롯에 언급됐으면 **그 슬롯의 핵심 표기를 그대로 이어 사용**.
3. 새로운 사실(시장 반응·주가)을 추가하는 건 OK. 단 이전에 "빅비트"라고 한 것을 morning 에서 "쇼크"로 뒤집으면 안 됨.

**금지**: 같은 사건을 pre 에서 "빅비트" 라고 한 뒤 morning 에서 "쇼크" 라고 정반대 표기 (= 2026-05-21 실제 사고)

### morning/afternoon 슬롯 본문 누락 금지

**발견**: 2026-05-21 morning(40자) / afternoon(40자) 가 헤드라인 한 줄만 emit. pre 466자 / post 499자 / evening 385자 와 비교해 비정상.

**규칙**:
- 모든 slot 은 본문 최소 **200자** 이상 (헤드라인 + 본문 + 체크 포인트 한 구조).
- skill 안에 `len(summary) < 200` 검증 추가하고 그 이하면 재생성 또는 에러.
- 헤드라인 한 줄만 만들고 본문 누락된 채 DB 저장하는 패턴은 사용자 신뢰를 직접 깨뜨림.

### 좋은 예 (2026-05-21 morning 정정 예시)

"어젯밤 엔비디아가 매출 816억\$·EPS 1.87\$ 로 컨센서스를 모두 상회하는 빅비트를 발표하면서 코스피가 +7.07% 사이드카 발동까지 갔어요. 사이드카는 분기에 한두 번 수준 폭등 신호예요. 오늘 오후 1시쯤 외인 누적이 +로 돌아오는지 한 번 확인해 주세요 — pre brief 에서 짚어드린 -19조 5일 누적 매도가 멈추는 첫 신호예요."

→ 본문 200자 이상, 이전 slot(pre)의 "빅비트" 표기 그대로 이어감, 시장 반응(사이드카 +7%)을 빅비트와 분리해 표현.

### 나쁜 예 (2026-05-21 morning 원본)

"엔비디아 어닝쇼크와 삼성 자사주 매입에 코스피 +7.07% 사이드카 발동"

→ (1) 40자 본문 누락, (2) "어닝쇼크" 사실관계 오류 (실제 빅비트), (3) pre 슬롯의 "빅비트" 표기를 뒤집음.

### 이미 발표된 사건 재예고 금지 (2026-05-26 강화)

**발단**: 2026-05-22 evening · 2026-05-26 pre brief 가 NVDA 어닝을 "다음 주 수요일 28일 발표 예정" 으로 적음. 실제 NVDA Q1 FY27 어닝은 **2026-05-20 새벽** 이미 발표돼서 (우리 5/21 pre brief 가 "매출 816억\$ 빅비트" 정확히 기록), 다음 차기 어닝은 8월. 사용자 피드백 인용: "이미 엔비디아랑 pce 저번주에 한걸로 아는데 아닌가"

**원인**: brief skill 이 "이번 주 핵심 이벤트" 를 외부 데이터 없이 작성 — MacroCollector 에 forward calendar 함수 부재, Claude 가 추측 또는 옛 brief 본문 그대로 복사.

**규칙**:

1. **이벤트 발표일 언급 전 — 이전 7일간 brief 에서 같은 이벤트 발표 기록 검증 필수**
   ```sql
   SELECT briefing_date, slot, substring(summary, 1, 200)
   FROM market_briefings
   WHERE briefing_date >= (current_date - INTERVAL '7 days')
     AND (summary LIKE '%엔비디아%' OR summary LIKE '%NVDA%' OR summary LIKE '%PCE%' ...)
   ORDER BY briefing_date DESC, slot_time DESC
   ```
   결과에서 "발표됐어요/빅비트/상회/하회" 같은 과거형 표현이 발견되면 그 이벤트는 **종료됨** — 미래 시제로 재언급하면 사실관계 오류.

2. **차기 어닝일은 분기 기준으로 확인**
   - NVDA Q1 FY27 (2026-05) → 다음 Q2 는 **2026-08월**
   - NVDA 차기 어닝을 "이번 주" 또는 "다음 주" 로 절대 표기 금지 (실제 발표 3개월 후)
   - 같은 분기 안에 같은 회사 어닝이 두 번 나오지 않는다는 상식

3. **매크로 지표 (PCE·CPI·고용지표) 일정은 외부 검증 후 작성**
   - PCE: 매월 마지막 영업일 또는 그 다음 (BEA 공식 캘린더). "X요일 발표" 라고 적기 전 web search 또는 외부 캘린더 확인.
   - "수요일 PCE" 같이 요일만 적는 것 금지 — **반드시 정확한 날짜 (예: "5월 30일 (금) PCE")**.
   - 추측으로 요일 박지 말 것 — 사용자가 외부 일정과 대조하면 즉시 발각.

4. **이번 주 이벤트가 정말 있는지 확신 없으면 침묵**
   - "이번 주는 큰 매크로 이벤트 없이 차분한 흐름이에요" 처럼 솔직히 짚는 게, 잘못된 일정을 박는 것보다 신뢰 깨지 않음.

### 나쁜 예 (2026-05-22 evening + 5/26 pre — 실제 사고)

"진짜 핵심은 다음 주 수요일 28일 PCE 와 엔비디아 실적이라 주말은 한 박자 쉬어 가셔도 되고..."
"수요일 PCE·엔비디아 전까지는 무리한 진입 자제해 주세요."

→ (1) NVDA 어닝은 5/20 발표 완료 (이미 우리 brief 가 빅비트로 기록), 5/28 발표 ❌ (2) PCE "수요일 28일" 검증 안 함 — 실제는 5/30 (금) (3) 두 brief 가 같은 잘못된 문구를 그대로 복사 → 사용자가 첫눈에 발각.

### 좋은 예 (정정안)

"엔비디아 빅비트(매출 816억\$)는 지난 주 5/20 발표돼 시장이 이미 반영했어요. 이번 주는 5/30 (금) 4월 PCE 발표 정도가 매크로 일정이라, 5월 28일 (수) 까지는 큰 이벤트 없이 종목 장 위주 흐름이에요."

→ (1) NVDA를 과거형으로 정확히 ("지난 주 5/20 발표돼...반영"), (2) PCE 일정을 정확한 날짜로 (5/30 금), (3) 이번 주 매크로 부재를 솔직히 표현.

### brief 작성 1단계 — 컨텍스트 자동 주입 (2026-05-26 신설)

모든 brief skill (pre/morning/afternoon/post/evening) 시작 직후 **반드시 한 번 호출**:

```bash
python scripts/brief_context_inject.py --slot <slot_name> --days-back 7 --days-ahead 10
```

4. **`top_news`** — 최근 36h 고신호 시장 뉴스 (news_events, 악재 우선·중복 dedup) — **2026-06-22 신설**
   - `items[]`: {timestamp, headline, sentiment(악재/호재), theme, reason}, `count_neg`/`count_pos` (그날 시장 무드)
   - **[배경] 사용자 피드백 인용**: "홈 브리핑에 눈에 보이는 대형 악재(이란-미국 전쟁 같은 국제정세)가 있는데 하나도 브리핑 안 함. 뉴스테이블에 다 있는데 전혀 안 읽는듯? 이러면 도움이 안돼."
   - **의무 규약**:
     - `items` 중 **지정학·거시 대형 악재**(전쟁·해협 봉쇄·환율 급변·관세·유가 급등 등)는 **headline 또는 첫 bullet 에 반드시 반영**. "홈 헤드라인 중요도 우선" 사다리의 1~2티어로 취급.
     - 단순 종목 뉴스가 아닌 **시장 전체를 흔드는 뉴스**가 우선 (호르무즈 봉쇄 → 유가·정유·해운·항공 / 환율 28년 최고 → 외인 수급).
     - "왜 떨어졌나" 인과 anchor 로 활용 (예: "외인 매도 -8천억 — 호르무즈 봉쇄로 유가 급등·환율 1,400 돌파가 위험회피 자극한 것으로 보여요").
     - `count_neg` 가 `count_pos` 보다 크게 많으면 그날 무드가 "악재 우위" — newbie_summary·headline 톤에 반영.
   - **🚫 재탕 금지 (2026-06-22 사용자 피드백 "너무 재탕하면 안돼")**:
     - top_news 는 최근 36h 라 **같은 뉴스가 여러 슬롯에 걸쳐 뜸**(호르무즈가 pre·morning·afternoon 다). 직전 슬롯이 이미 다룬 뉴스를 **같은 문장·같은 각도로 복붙 금지**.
     - `recent_briefs` 로 직전 슬롯이 그 뉴스를 어떻게 다뤘는지 확인 → **새 전개·진전만** 추가 (오전 대비 시장 반응, 밤 사이 변화, 해소/악화 여부). 진전 없으면 한 줄로 줄이거나 생략.
     - 예: pre 가 "호르무즈 봉쇄로 유가 급등 우려" → afternoon 은 "(재탕 ❌) 오전 우려대로 정유주 +5% / 항공 -3% 로 갈렸어요 (✅ 진전)".
     - 며칠 이어지는 악재는 **카운트다운·누적 관점**으로 (예: "호르무즈 긴장 사흘째, 유가 76→79달러"), 매번 똑같이 "봉쇄됐어요" 반복 금지.
   - **금지**: top_news 에 대형 악재가 떠 있는데 코스피 지수·상한가 얘기로만 brief 채우기 (= 2026-06-22 사고).

**핵심 효과**:
- "수요일 28일 PCE 와 엔비디아 실적" 같이 **이미 끝난 사건 + 진짜 일정 묶기** 사고 차단
- NVDA 어닝일을 추측하지 않고 calendar 보고 작성 → "다음 10일에 NVDA 없음" 이면 brief 에 안 적음
- 빅 이벤트 (스페이스X 같은 대형 IPO·빅테크 어닝·매크로 발표) 누락 방지
- 직전 slot 표기 일관성 강제 (어닝 쇼크 vs 빅비트 사고 차단)

### 이벤트 중요도별 본문 풀이 의무 (2026-05-26 신설)

`brief_context_inject` 가 반환한 `mega_events` 배열의 각 항목은 size 에 따라 brief 본문에서 **반드시 다음 분량으로 다뤄야 한다**. 그냥 "~예정이에요" 한 줄로 넘기면 안 됨.

| size | brief 본문 분량 | 형식 |
|---|---|---|
| **mega** | **최소 한 단락 (100-200자)** | 사실 + 한국 시장 영향 + 관찰 단서. 별도 줄바꿈으로 분리해 강조 |
| **high** | **한 문장 (40-80자)** | 일정 + 한국 시장 영향 한 줄 |
| **medium** | 짧은 한 줄 또는 생략 가능 | 체크 포인트 bullet 안에 한 마디 |
| **low** | 생략 | 본문에 안 넣음 |

### mega 이벤트 풀이 형식 (mandatory)

```
<강조 도입 — "이번 주 가장 큰 이벤트 / D-N 다가옴">
<사실 1-2 문장 — 어디서·언제·규모·핵심 숫자>
<한국 시장 영향 1-2 문장 — 누가 수혜·누가 압박·관찰 단서>
```

**좋은 예 (mega — 스페이스X IPO)**:

> 이번 주 시장 최대 이벤트는 6/12 (금) 스페이스X 나스닥 상장이에요 — 티커 SPCX. 목표 시가총액 1.75조 달러로 MS 를 능가하고 애플·엔비디아 다음 가는 규모이고, 공모 규모만 최대 750억 달러(약 100조원)로 역대 최대 IPO 예요. 글로벌 펀드가 SPCX 청약·후속 매수 위해 다른 시장 비중 줄일 가능성이 있어서 한국 외인 수급이 6월 첫 주 변동성 커질 수 있어요. 반대로 위성·우주항공 테마는 단기 모멘텀 — 한화에어로·KAI·쎄트렉아이가 첫 30분 갭상승하는지 봐 주세요.

**나쁜 예 (mega 인데 한 줄로 끝)**:

> 6월 중 스페이스X IPO 예정이에요.

→ mega 인데 200자 미만 + 한국 시장 영향 누락 = 룰 위반. 본문 길이 부족 검증으로 재생성 또는 사용자 피드백 발각.

### high 이벤트 풀이 형식

```
<일정 + 한 줄 영향>
```

**좋은 예 (high — NVDA Q2 어닝)**:

> 8/20 (수) 새벽 NVDA Q2 어닝 발표 예정. 매출 가이던스 컨센서스 상회 여부가 한국 반도체 (SK하이닉스·삼성전자) 8월 후반 흐름을 좌우해요.

### 의무 규칙

| size | brief 본문 포함 임계 | 처리 |
|---|---|---|
| **mega** | `d_minus ≤ 30` (한 달 전부터) | 단락 풀이 (100-200자) 의무 |
| **high** | `d_minus ≤ 14` (2주 전부터) | 한 문장 의무 |
| **medium** | `d_minus ≤ 7` (1주 전부터) | 체크 포인트에 한 줄 |
| **low** | (생략) | 본문 미포함 |

1. mega 는 한 달 전부터 미리 알린다 — "스페이스X IPO D-17 다가옴" 같이 카운트다운 톤으로. 사용자 피드백 인용: "스엑은 역대 최대규모 상장 블랙홀 — brief 에서 안 다루네".
2. mega 이벤트가 2개 이상이면 가장 임박한 1개에 집중 + 나머지 한 줄씩.
3. 이벤트가 끝났으면 (`d_minus < 0` 또는 status=completed) 다음 brief 부턴 과거형으로만 회상.
4. `big_events.json` 항목의 `facts_kr` / `impact_kr` 본문이 있으면 그 톤·내용을 brief 카피로 자연스럽게 풀어 옮긴다 (단순 복붙 X, 주린이 카피 규약 준수).
5. `mega_events` 항목의 `_source: "manual_big_events"` 는 운영자가 직접 검증해 등록한 사실. 자동 detect (`_source: "finnhub_auto"`) 보다 우선 다룬다.
